# 콘텐츠 엔진 설계 — 1 주제 → 1 정본 → N 채널 (OSMU 파이프라인)

> 작성 2026-07-08 · 상태: **Phase 0 설계(구현 전)**
> 상위 전략: 메모리 `bideasy-content-strategy` · 발행 캘린더: `docs/CONTENT_CALENDAR.md`(주제·리듬 정본) · 발행 인프라: `docs/BLOG_RUNTIME_PUBLISHING.md`
> 이 문서는 **"어떻게 하나의 소스로 블로그·인스타·유튜브를 다 만드나"**의 아키텍처 정본이다. 주제 목록은 캘린더가, 발행 타이밍은 `publish_at` 스케줄러가 담당하고, 이 문서는 **그 사이의 생성·파생 구조**를 정의한다.

---

## 0. 결론 먼저

- **문제 재정의**: "블로그를 꾸준히"가 아니라 **"하나의 콘텐츠 소스로 모든 채널을 먹인다"**. 채널마다 0에서 만들면 지속 불가능.
- **핵심 설계 결정**: 콘텐츠를 **산문 한 덩어리가 아니라 "구조화 블록"**(훅·요약·핵심·데이터·CTA)으로 생성한다. 블로그·카드뉴스·릴스·유튜브는 **전부 이 블록의 렌더 결과**다. 이 구조를 안 잡으면 채널 파생마다 사람이 다시 쪼개야 한다.
- **자동화 경계(정직)**: **텍스트 산출물 = 자동**(블로그 본문 + 카드 카피 + 릴스/유튜브 대본), **시각물 = 반자동**(도구 렌더), **채널 업로드 = 사람 최종 확인**. 완전 무인은 브랜드(정밀·정직) 리스크라 **검수 게이트 유지**.
- **이미 있는 것과의 접점**: 데이터스토리(`data_story.py`)는 이미 구조화 조각(30초 요약·TOP10 표·인사이트)을 뽑는다 → 이 스키마로 일반화. 카드 렌더는 `/cardmaker`(1080×1350, 힉스필드 배경) 재사용. 발행은 방금 만든 `publish_at` 유예·예약 스케줄러로 자동 드립.

## 1. 아키텍처 — 4 레이어

```
[L1 주제 큐]  상록수(A) + 입찰상식(K) + 데이터스토리 슬롯(B)
     │  우선순위·타겟검색어와 함께 (캘린더 §Track A / 이 문서 §4 / 캘린더 §Track B)
     ▼
[L2 소스 엔진]  주제 → 구조화 정본(ContentSource: 훅·요약·핵심·데이터·CTA)
     │            └ 데이터스토리는 숫자를 DB에서 결정적으로, 서술만 LLM (환각 차단)
     │  ★ 사람 검수 1회(법·숫자) — 여기서 승인해야 아래로 흐름
     ▼
[L3 채널 파생]  정본 블록에서 각 채널 산출물을 변환 생성
     ├─ 블로그 정본     blocks → body_md (SSR·OG·JSON-LD·sitemap = 기존 파이프라인)
     ├─ 인스타 카드뉴스  요약+핵심+데이터 → 슬라이드 카피(JSON) → /cardmaker 렌더
     ├─ 인스타 릴스/숏폼  훅+핵심 → 3초 훅 대본(15~30초)
     └─ 유튜브 롱폼     본문 → 말하기체 대본 + 챕터 + 설명란 + 썸네일 문구
     ▼
[L4 발행·배포]  블로그 = publish_at 자동 드립 / 외부 채널 = 반자동(렌더 후 사람 업로드)
     └ 모든 채널 말미 → bideasy.kr 회수(무료 계산기·자격 진단·14일 체험) + 크로스링크
```

## 2. 구조화 정본 스키마 (ContentSource) ★핵심

모든 채널이 이걸 읽는다. **채널이 아니라 소스가 정본.**

| 블록 | 내용 | 어느 채널이 쓰나 |
|---|---|---|
| `track` | `evergreen`(상록수) \| `knowledge`(입찰상식) \| `data_story` | 파생 규칙 분기 |
| `title` / `target_keyword` | 제목 · SEO 타겟 검색어 | 블로그(SEO)·유튜브 제목 |
| `hook` | 1~2문장 호기심 훅 ("왜 100개사가 몰려도…") | 릴스 3초·유튜브 인트로·카드 표지 |
| `summary_30s` | 30초 요약(2~3문장 또는 불릿) | 블로그 상단·카드 2p·릴스 |
| `key_points[]` | 핵심 3~5 (`{heading, body}`) | 블로그 본문·카드 슬라이드·유튜브 챕터 |
| `data_blocks[]` | (선택) `{caption, table|numbers, source}` — 우리 개찰데이터 | 카드(랭킹·숫자 최적)·유튜브 그래픽 |
| `internal_links[]` | /calculator·/diagnose·/search 등 | 전 채널 |
| `cta` | bideasy.kr 회수 문구 | 전 채널 말미 |

**렌더 원칙**: 블로그 `body_md`는 이 블록에서 **생성**한다(역방향 아님). 즉 블록이 원본, 블로그도 파생. 이렇게 해야 채널 간 메시지가 안 어긋난다.

### 데이터스토리 정합 (기존 코드 재사용)
`data_story.build_weekly_story()`가 이미 `summary`(30초)·TOP10 표·insight를 만든다 → 이를 위 스키마(`hook`=최다경쟁 훅, `data_blocks`=TOP10/기회공고 표, `key_points`=인사이트)로 매핑만 하면 같은 파생 라인에 태울 수 있다. **숫자는 계속 DB 결정적, 서술만 LLM** 원칙 유지.

## 3. 채널 파생 규칙

| 채널 | 입력 블록 | 산출물(자동) | 렌더/업로드 |
|---|---|---|---|
| **블로그**(정본 허브) | 전체 | `body_md`+`body_html` | `publish_at` 자동 발행 |
| **인스타 카드뉴스** | `hook`·`summary_30s`·`key_points`·`data_blocks` | 슬라이드별 카피 JSON(표지→핵심 3~5→데이터→CTA) | `/cardmaker` 반자동 렌더 → 사람 업로드 |
| **인스타 릴스/숏폼** | `hook`·`key_points`(2~3) | 15~30초 대본(3초 훅·핵심·CTA) | 외부 영상툴 → 사람 업로드 |
| **유튜브 롱폼** | 전체 | 말하기체 대본 + `chapters[]` + 설명란 + 썸네일 문구 + 자막 | 외부 영상툴/TTS → 사람 업로드 |
| **네이버 블로그**(요약형) | `hook`·`summary_30s`·`key_points` | 요약 본문 + "전문 보기" 원문 링크 | 사람 업로드(네이버 도달, §9.1) |

- **카드뉴스 톤 고정**(캘린더 §OSMU): 힉스필드 딥블루 배경, 미감보다 **숫자·실익 중심**.
- **랭킹·숫자형(데이터스토리)** → 카드뉴스 최적. **개념·오해형(입찰상식)** → 릴스/유튜브 훅 최적. 파생 우선순위를 `track`로 자동 판단.

## 5. 자동화 경계 (완전 무인 금지 지점 명시)

| 산출물 | 자동 | 반자동 | 사람 |
|---|---|---|---|
| 구조화 정본(블록) | ✅ LLM 생성 | | ★ **검수 1회(법·숫자)** — 필수 게이트 |
| 블로그 본문/HTML | ✅ 블록→렌더 | | 검수에 포함 |
| 카드 슬라이드 카피 | ✅ | 이미지 렌더=/cardmaker | 업로드 |
| 릴스/유튜브 대본·챕터·설명란 | ✅ | 영상 렌더·TTS=외부 도구 | 업로드 |
| 채널 게시 | | | ✅ 최종 확인(계정·브랜드 안전) |

- **검수 게이트는 1회로 집중**: 정본 승인 시점에 법·숫자·톤을 본다. 이후 파생은 그 승인된 블록에서 나오므로 채널별 재검수는 가벼운 눈대중.
- **데이터스토리는 예외적으로 유예 자동발행**(숫자 결정적) — `BLOG_AUTOPUBLISH_GRACE_HOURS`. 입찰상식·상록수는 **사람 승인 후 드립** 권장(법·숫자 서술 비중 큼).

### 5.1 시각물 생성 규칙 (2026-07-10 실측 확정) ★

> 검증: 힉스필드 `nano_banana_pro`로 "예정가격 추첨 흐름" 인포그래픽 1장 테스트 생성 →
> **한글 라벨 5종·숫자 3종(15개·4개·±2%) 전부 정확 렌더.** 브랜드 블루·플랫 벡터 품질도 즉시 사용 수준.
> "생성모델은 한글 도식 못 만든다"는 가정은 기각 — 도식도 생성으로 간다.

- **도식·인포그래픽 = 힉스필드 `nano_banana_pro`** (텍스트·다이어그램 특화 모델 명시 사용). 단 **생성 후 사람 텍스트 눈검수 필수** — 글자·숫자 하나라도 틀리면 재생성(생성물은 매회 100% 보장 아님, '정확=브랜드').
- **히어로·분위기컷 = 힉스필드** — 브랜드 딥블루, **텍스트 미포함**(은유·개념 이미지).
- **텍스트 위주 카드뉴스 = `/cardmaker` 병행** (기존 딥블루 톤 유지).
- **호스팅**: 생성물 다운로드 → `/assets/blog/{slug}/` 배치 → 마크다운 `![캡션](경로)` — 캡션이 figcaption+alt로 들어가 SEO·접근성 자동 확보. **외부 CDN URL 직접 참조 금지**(의존성·수명 리스크).
- **삽입량**: 글당 히어로 1 + 도식 1~2가 적정(lazy-load 적용됨, 과다 삽입은 산만·로딩 저하).
- **프롬프트 요령**: 렌더할 한글 문구를 따옴표로 **정확히 명시**("Render the Korean text labels EXACTLY: '…'") + 브랜드 팔레트 hex + flat vector/no photorealism.

## 9. SEO — 엔진 설계에 반영 (블로그 = 검색 유입 토대)

> 현황: 블로그 SSR 이미 `Article`+`datePublished`/`dateModified`(`blog_detail.html`), 목록 `BlogPosting`(`blog_list.html`), canonical·www 301·sitemap·OG 확보. Google/Naver 소유확인+sitemap 제출 완료(2026-06-16). 아래는 그 위에 **엔진이 지켜야 할 규칙**.

### 9.1 네이버·구글 이원화 (가장 중요)
타깃(40~60대 시공사)은 **네이버 검색 비중이 큰데, 네이버는 자사 생태계(블로그·카페)를 우대**해 bideasy.kr 단독 오가닉 상위는 어렵다. 반면 구글은 우리 SSR 허브가 잘 먹힌다.
- **구글** = `bideasy.kr/blog` 정본 허브(현행 최적화 유지).
- **네이버** = **네이버 블로그 재발행을 정식 채널 파생으로.** 단 통째 복붙은 자기잠식(중복) → **요약형 + "전문 보기" 원문 링크**(부분 재발행). 원문에 canonical 유지.
- L3 채널 파생 표에 `네이버 블로그(요약형)` 행 추가.

### 9.2 자동생성 콘텐츠 — 구글 스팸정책(scaled content abuse) 방어
대량 자동발행은 페널티 리스크. 우리 방어책을 **설계 불변식**으로 고정:
- **검수 게이트**(상식·상록수 = 사람 승인 후 발행) + **복제 불가 개찰데이터**(스핀 아님) + **저자 엔티티**(BidEasy).
- ⚠️ **데이터스토리 주간 페이지(`weekly-YYYY-wNN`) 관리**: 얇거나 서로 유사하면 doorway/중복으로 찍힘 → **최소 데이터 임계**(이미 `no_data` skip)**·매주 고유 앵글·상호 내부링크** 필수. 빈약한 주는 발행하지 않는다.

### 9.3 구조화 데이터 매핑 (블록 설계의 SEO 이점)
정본 블록 → 스키마가 자동으로 따라온다:
- **입찰상식(K, Q&A형) → `FAQPage`** (검색 리치 결과 노출).
- **상록수 절차 가이드(계산법 등) → `HowTo`**.
- **유튜브 임베드 글 → `VideoObject`**.
- 기존 `Article`+`dateModified` 유지 — **자동발행 시 `date`/`updated`가 실제 갱신되는지 확인**(최신성 신호).

### 9.4 유튜브 = 제2 검색엔진 (구글 소유)
롱폼은 구글 SERP·유튜브 검색 양쪽 노출. 파생 산출물에 **대본 + 챕터(타임스탬프→구글 key moments) + 설명란 키워드 + 자막(transcript)** 포함(자막은 색인 대상).

### 9.5 키워드 규율 + 토픽 클러스터
- **1글 = 1 주검색어**: 제목·H1·URL 슬러그·`summary_30s`·첫 문단에 일관 배치(캘린더가 타겟검색어 배정).
- **허브-스포크**: 상록수 pillar(입찰 시작법·적격심사)에 입찰상식 스포크를 `internal_links` 블록으로 묶어 클러스터 형성. 기초 4편이 허브.

### 9.6 기술적 체크(확보됨 — 확인만)
